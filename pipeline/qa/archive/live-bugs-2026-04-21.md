### AS-3 — Custom-field column values cannot be edited after the column is added

- Status: `[x]` — merged in `collinear-ai/Asana-Clone#10` after orchestrator browser verification on `localhost:3002`.
- Repro:
  1. Log in as `sarah.connor / password` on `http://localhost:3002`.
  2. Open any project (e.g. `prj_001`).
  3. Click `+` in the list header → **Create new field** → type a name,
     **Create**.
  4. Try to type or pick a value in any row of the new column.
- Observed: The cell opens a dropdown popover that only shows `—` (no
  selectable options). The column is therefore dead on arrival.
- Root cause:
  - `ProjectListView.handleCreateFieldForProject`
    (`app/frontend/asana-clone/src/components/features/projects/ProjectListView.tsx:418`)
    hardcodes `addCustomField(name, 'dropdown', [])` — the created field
    is always a `dropdown` with zero options.
  - `CustomFieldsPage.handleCreate`
    (`app/frontend/asana-clone/src/components/features/customfields/CustomFieldsPage.tsx:29`)
    does the same.
  - The "Create new field" popover has no type selector and no options
    editor.
  - `store.updateCustomField`
    (`app/frontend/asana-clone/src/data/store.ts:1202`) only supports
    renaming — no `addOption` / `removeOption` helpers — even though the
    backend `app/tools/custom_fields.py` already exposes `add_options` /
    `remove_option_ids`.
- Proposed fix:
  - Add a **type picker** (`text`, `number`, `dropdown`, `date`, `person`)
    and, when `dropdown` is selected, an inline **options editor** to the
    "Create new field" popover in both `ProjectListView` and
    `CustomFieldsPage`.
  - Default the type to `text` so a new column is immediately editable.
  - Add `addOptionToField` / `removeOptionFromField` to `store.ts`, wired
    to the existing backend endpoints.
  - Surface a "+ Add option…" row inside `CustomFieldValueCell`'s
    dropdown editor so dropdown fields created in the past (like the
    current `dddd`) can be fixed without recreating them.
- Affected files:
  - `app/frontend/asana-clone/src/components/features/projects/ProjectListView.tsx`
  - `app/frontend/asana-clone/src/components/features/customfields/CustomFieldsPage.tsx`
  - `app/frontend/asana-clone/src/components/common/CustomFieldValueCell.tsx`
  - `app/frontend/asana-clone/src/data/store.ts`
  - `app/frontend/asana-clone/src/data/types.ts` (only if the store
    signature changes)

## Notion

### NT-5 — Editing a row title opens a popover instead of inline text input

- Status: `[x]` — merged in `collinear-ai/notiondesktop-clone#7` after orchestrator browser verification on `localhost:3004`.
- Repro:
  1. Log in on `http://localhost:3004`.
  2. Open a database page (e.g. `page_003` — Project Tracker).
  3. Click a row's **Title** cell (e.g. "Add search functionality").
- Observed: A floating `PropertyValueEditor` portal popover opens
  *below* the cell with a single-line text input. The cell itself never
  becomes editable in place.
- Root cause:
  - `TableView.tsx:566-577` wires the title-cell `onClick` →
    `setEditing({ entry, property })`.
  - When `editing` is set, `TableView` renders
    `<PropertyValueEditor ... />` which is a `createPortal`-based popover
    (`PropertyValueEditor.tsx:94`).
  - For the `title` property, `EditorBody` routes to `TextLikeEditor` —
    an input inside that portal popover — instead of rendering the input
    directly inside the `<td>`.
- Proposed fix:
  - In `TableView.tsx` only, for the **title column**:
    - when the cell is in editing mode → render an `<input autoFocus>`
      *inside the `<td>`* (no portal), with `onBlur`/Enter →
      `updateEntry(entry.id, { [titleProp.id]: draft })` and Escape →
      cancel.
    - when not editing → keep the current read-only label.
  - Skip the portal `PropertyValueEditor` render for the title property
    only. Keep portal behaviour for select / date / person (those
    genuinely need popover UI).
- Affected files:
  - `app/frontend/src/components/TableView.tsx`

### NT-6 — Inline "Add a comment…" opens the side drawer instead of an inline composer

- Status: `[x]` — merged in `collinear-ai/notiondesktop-clone#8` after orchestrator browser verification on `localhost:3004` (typed "Inline composer works via NT-6", Comment button submitted inline, composer reset to stub, comment then visible in the side Comments panel).
- Repro:
  1. Log in on `http://localhost:3004`.
  2. Open any page in full view.
  3. Scroll to the bottom of the properties block → click
     **Add a comment…**.
- Observed: The right-side `CommentsPane` drawer opens and the textarea
  lives *in that drawer*, not inline in the page body.
- Root cause:
  - The NT-4 fix (PR #6 on `notiondesktop-clone`) wired
    `CommentsStub.onClick` → `onAddComment` →
    `App.setCommentsOpen(true)` → right-side drawer.
  - User wants an inline composer in the page body instead.
- Proposed fix:
  - Convert `CommentsStub` in `PagePropertiesList.tsx` to own a local
    `composing` state. When toggled on, render a `<textarea>` + a
    **Comment** / **Cancel** pair in the same place, wired directly to
    `createComment(pageId, { content: [{ text }] })` (already exported
    from `api.ts` and used by `CommentsPane`).
  - Keep the right-side `CommentsPane` untouched — still reachable via
    the page header Comments icon.
  - `onAddComment` stays in the prop API but is no longer required;
    surfaces that still want the old drawer behaviour (e.g. the row peek
    drawer) can pass it explicitly.
- Affected files:
  - `app/frontend/src/components/PagePropertiesList.tsx`

### NT-7 — Page comments should render as an inline feed, and the `All discussions` drawer should be full-page-only

- Status: `[x]` — merged in `collinear-ai/notiondesktop-clone#9` after
  orchestrator browser verification on `localhost:3004` in **both**
  views:
  - Full page (`page_008`): inline feed rendered existing comments with
    author + relative time under the properties block, composer posted
    a new comment, feed auto-refreshed. Topbar Comments toggle still
    opens/closes the right-side `All discussions` drawer as before.
  - Peek drawer (opened from a database row on `page_003`): inline feed
    rendered identically under the properties, composer posted and
    auto-refreshed, and the right-side `All discussions` drawer stayed
    closed (peek top bar intentionally has no Comments toggle).
- Repro (based on the two reference screenshots from the owner):
  1. Log in on `http://localhost:3004`.
  2. Click a database row title to open the peek drawer.
  3. Scroll to the bottom of the properties block.
- Observed (pre-fix):
  - In peek, the properties block ended with a bare `Add a comment…`
    stub — existing page comments were **not** surfaced inline.
  - In full page, the side `All discussions` drawer was available from
    peek too (a full-page-only affordance in the reference).
- Intent:
  - **Image 1**: page comments should appear as an inline **feed**
    under the properties, in both views (not only the drawer).
  - **Image 2**: the right-side `All discussions` drawer should be a
    full-page-only affordance — hidden (and auto-closed) in peek.
- Proposed fix:
  - In `PagePropertiesList.tsx`, extend `CommentsStub` to also hold a
    `feed` loaded via the existing `fetchComments(pageId, false)` API,
    filtered to top-level page-scoped unresolved comments. Render each
    as a new `InlineCommentItem` (avatar initial + author + relative
    time + text). Refresh the feed after a successful
    `createComment(...)` in the inline composer.
  - In `App.tsx`, add a `useEffect` that calls `setCommentsOpen(false)`
    whenever `peekPageId` becomes truthy, so opening a peek
    automatically collapses the side drawer (which peek's top bar
    can't reopen anyway).
- Affected files:
  - `app/frontend/src/components/PagePropertiesList.tsx`
  - `app/frontend/src/App.tsx`
